MCP App Renderer — Design Overview

Phase 1 scope: what we're building now vs. what comes later

Phase 1: Ship Now
MCP App HTML (monolithic bundle)
include_str! in runt-mcp
React MIME dispatch ANSI renderer text/plain text/html images JSON errors streams markdown plugin KaTeX plugin ext-apps bridge
Data flow
host → ext-apps SDK → ontoolresult
→ structuredContent (cells + blob URLs)
→ React renderer (MIME priority dispatch)
→ fetch blob URL → render with plugin
→ images: <img src="blob_url"> (no fetch)
Fixes
✓ No "Connected — waiting for execution..."
✓ No raw blob URLs ever shown
✓ Markdown + KaTeX rendered properly
✓ Same React renderer as notebook app
✓ ~1.2MB bundle (React + markdown + KaTeX)
Phase 2: Later
Daemon HTTP plugin serving
http://localhost:{port}/plugins/{name}.js
plotly (~4MB) vega (~2MB) leaflet (~200KB) parquet/WASM
How it works
renderer sees plotly MIME type
→ fetch("{blob_base_url}/plugins/plotly.js")
→ installRendererPlugin(code, css)
→ render with plotly
Daemon changes needed
• Add /plugins/ route to daemon HTTP server
• Serve pre-built CJS plugin bundles
• Same include_str! build, different serving path
• WASM: serve .wasm alongside .js loader
Architecture contract
installRendererPlugin(code, css?)
Same interface whether code comes from include_str!, Vite virtual module, or HTTP fetch
