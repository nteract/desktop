MCP Apps Output Rendering: Current vs. Proposed

How should the MCP App widget render cell outputs?

Current: Vanilla DOM (mcp-app.ts)
ext-apps SDK → ontoolresult → vanilla DOM
✓ Works
• Images (blob URL → <img>)
• HTML tables (srcdoc iframe)
• Text/plain, JSON
• Stream, error outputs
✗ Broken
• Plotly charts → raw blob URL text
• Vega/Vega-Lite → raw blob URL text
• GeoJSON/Leaflet → raw blob URL text
• Markdown → plain text
• LaTeX → plain text
• "Connected — waiting for execution..."
Proposed: Embedded Renderer Bundle
ext-apps SDK → ontoolresult → React renderer + plugins
✓ Everything works
• Same renderer as notebook app
• Plotly, Vega, Leaflet plugins
• Markdown + KaTeX rendering
• Images, HTML, JSON, text
• MIME priority dispatch
• ANSI color rendering
Key changes
• Bundle React + renderer into MCP App
• Lazy-load plugins on first viz output
• structured.rs sends inline viz specs
• No blob URLs ever shown to user

Pick an approach

A

Patch the vanilla widget

Keep mcp-app.ts as vanilla DOM. Fetch viz spec blob URLs and render with lightweight inline libs. Fix blob URL display and boilerplate text. Ship fast, maintain two codepaths.

B

Embed the renderer bundle (recommended)

Replace vanilla rendering with the React isolated renderer + plugin system. One codepath for notebook and MCP Apps. structured.rs sends inline viz specs instead of blob URLs. Bigger upfront but eliminates divergence.

C

Nested isolated iframes

MCP App creates child IsolatedFrame components per output. Maximum code reuse (literally same component). But double-iframe nesting may hit host CSP issues and adds complexity.
