# Claude Code on the Web: Cloud Session Bootstrap

This repo wires two scripts so cloud Claude Code sessions (claude.ai/code) can read everything correctly and build the workspace. Local sessions are unaffected — both scripts gate on `CLAUDE_CODE_REMOTE=true`, which Anthropic sets only in cloud VMs.

## What works in cloud sessions

- **Read everything correctly.** LFS-tracked plugin bundles (`.wasm`, `.js`, `.css`) appear as real content, not pointer files. Tools like `Read`, `Grep`, ultrareview, ultraplan all see source code.
- **Build the workspace.** `cargo check`, `cargo build`, `cargo xtask build --rust-only`, `cargo xtask wasm` all succeed. `cargo test` works for any crate that doesn't require LFS-only fixtures.

## Out of scope

- Running the daemon (`cargo xtask dev-daemon`) — needs deno for kernel bootstrap.
- Python integration tests (`cargo xtask integration`) — needs maturin, venvs.
- The desktop app (`cargo xtask notebook`) — needs Tauri system deps + a display.
- The two `.arrow` test fixtures under `packages/sift/public/` — those are LFS-only (no source to regenerate from). Sift-package tests that require them will fail in cloud.

If a workflow above is needed, extend `scripts/cloud-setup.sh` and `scripts/cloud-bootstrap.sh`.

## How it works

Two layers, picked from Anthropic's [cloud sessions documentation](https://code.claude.com/docs/en/cloud):

| | Setup script | SessionStart hook |
| --- | --- | --- |
| Where it's configured | Cloud-env UI at claude.ai/code | `.claude/settings.json` (in repo) |
| When it runs | Once per env snapshot (~7 days) | Every session start and resume |
| Caches as a snapshot | Yes | No |
| Runs as | root | Same user as Claude |
| In this repo | `scripts/cloud-setup.sh` | `scripts/cloud-bootstrap.sh` |

`scripts/cloud-setup.sh` installs the tools the Anthropic image is missing (`git-lfs`, `wasm-pack`) and warms the cargo registry plus pnpm store. The output snapshots into the cached environment, so subsequent cold sessions skip the install cost.

`scripts/cloud-bootstrap.sh` runs every session. It refreshes `pnpm install`, regenerates LFS-tracked build artifacts (see below), and warms cargo. All steps are idempotent — fast no-ops when caches are warm.

## Why we regenerate LFS artifacts instead of `git lfs pull`

The Anthropic sandbox cannot reach Git LFS object storage:

- The local git proxy returns HTTP 502 on `/info/lfs/objects/batch`.
- The actual LFS object hosts (`media.githubusercontent.com`, `github-cloud.githubusercontent.com`) are blocked by the sandbox network policy.
- The MCP `get_file_contents` tool also returns the LFS pointer text.

So `git lfs pull` and any HTTP-API fallback both fail. Instead, the bootstrap detects pointer files in the checked-in artifact paths and runs `cargo xtask wasm`, which rebuilds the WASM (`runtimed-wasm`, `sift-wasm`) and chains into the renderer-plugin bundles. The regenerated bytes aren't byte-identical to the LFS originals (build IDs, minification timestamps differ), but they're functionally equivalent — `include_bytes!` size assertions pass, frontend tests parse real JS, and the agent reads real source.

Do not "fix" this back to `git lfs pull` without confirming the sandbox network policy has changed.

## Configuration

### One-time setup (per cloud environment)

In the cloud-env UI at claude.ai/code, set the **Setup script** field to this literal one-liner:

```bash
bash scripts/cloud-setup.sh
```

That's it. All real logic lives in repo, not in the UI field.

If you don't configure this, sessions still work but every cold session pays the install cost via the bootstrap fallback (slower, no caching benefit).

### Per-session bootstrap (already wired)

`.claude/settings.json` has a `SessionStart` hook that runs `scripts/cloud-bootstrap.sh` on every cloud session. No setup needed beyond the one-liner above.

## Debugging

The bootstrap writes a full transcript to `/tmp/cloud-bootstrap.log` on every session. If something looks off, ask Claude to `Read` that file.

The bootstrap always exits 0 — a failure never blocks the session. The agent can read the log and recover (e.g., manually run `pnpm install` or `cargo xtask wasm`).

Common symptoms:

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| `Read` on `*.js` plugin file shows `version https://git-lfs...` | wasm regen didn't run | Check `/tmp/cloud-bootstrap.log` — if `wasm-pack` is missing, configure the setup script |
| `cargo build` fails with `assertion failed: bytes.len() > 1024` on `include_bytes!` | LFS pointer bytes still in place | Run `cargo xtask wasm` manually, then rebuild |
| Bootstrap reports `pnpm install: FAILED` | Lockfile drift or proxy issue | Check the log; try `pnpm install --force` |

## Future work

- Pre-baking real LFS artifacts into the snapshot (publish them as a GitHub release asset and pull from `release-assets.githubusercontent.com`, which is on the default Trusted allowlist) would eliminate per-session `cargo xtask wasm` cost. Worth considering once cloud usage justifies the maintenance burden of cutting an asset release on every WASM-affecting PR.
- Adding deno + uv venv setup unlocks `cargo xtask integration` (tier iii) for cloud test runs. Skip until a concrete cloud workflow needs it.
